> Edward 6/27 拍板 V1 兩引擎完整上線後，卡西法讀完 5 份文件（GPU 經濟學 / Ditto 串流定論 / 引擎現況 / 商業點數 / 系統骨架）產出。給 codex 拆工項用。
> 配套：成本驗算見 `商業模式-v3-成本現實與訂價修正-2026-06-27.md`。

## ⚠️ 開工前三個前提校正（把鎖死的商業前提翻成技術坑）

**校正 1 · 兩引擎成本性質不對稱、架構要吃這不對稱**
- Ditto（標準）= 1×RTX4090、scale-to-zero 可真.睡死零成本（serverless GPU 按秒計費）。
- LiveAvatar（生動）= 5×H800 一組、**單組只服務 1 路 45fps**；「醒來」是 5 卡同起、**一組只能服務一個尊榮用戶**。
- → 生動引擎不是「一個 pool 服 N 人」，是 **per-session 獨佔資源租借**。**若 codex 照 Ditto 的「共享 pool」邏輯蓋生動引擎，上線必爆。這是全藍圖最關鍵架構分歧點。**

**校正 2 ·「天生吃兩引擎」≠「兩引擎程式碼長一樣」**
- 相同的是**外殼合約**、不是引擎內部。正解＝定義一個 `AvatarEngine` 抽象介面、兩引擎各自實作。外殼（路由/計量/串流/多租戶）只認介面、不認引擎。第三個引擎進來時外殼一行不改。**這才是「一次蓋對」的工程意義。**

**校正 3 · 三 keystone 未證 = 藍圖必須「容錯設計」非「假設成功」**
- (a) Ditto TRT 25fps、(b) LiveAvatar 45fps、(c) 冷啟動秒數 — 全 unknown。每個元件都標「若 keystone 失敗怎麼退」、不容許假設都過就蓋死。

## A · 兩引擎共用的雲端外殼（AOL · Avatar Orchestration Layer）

### A.0 核心抽象：`AvatarEngine` 介面（codex 第一個蓋的東西）
```
interface AvatarEngine:
  warmup() -> ready_signal            # 喚醒 GPU、回報就緒（含冷啟動）
  open_stream(session_id, face_id, voice_track) -> stream_handle
  push_audio_chunk(stream_handle, pcm_chunk) -> video_frames[]   # 串流：音進影格出
  idle_frame(face_id) -> video_frame  # 待機 idle-loop 影格
  close_stream(stream_handle)
  health() -> {fps_actual, first_frame_latency_ms, gpu_util}
  cost_meter() -> {gpu_seconds_consumed}   # 給計量層
```
- DittoEngine（1×4090、TRT 路）、LiveAvatarEngine（5×H800 獨佔組）各自實作；外殼只呼叫介面。
- 價值：LiveAvatar 未驗時可先用 stub（回傳假影格）頂著、外殼照樣整合測試。

### A.1 待機 idle-loop（卡西法 PoC verdict、鎖死）
- 長輩進聊聊未開口 → 播**本地預生 idle loop 影片**（眨眼呼吸微笑、3-5 秒無縫循環、存 CDN、App 本地播）→ **GPU 完全不啟動**。
- 開口（VAD 偵測）/ 寧寧回話 → 才 warmup 喚 GPU → open_stream → 串流生「說話那幾秒」。
- 說完回沉默 → 關串流、無縫切回 idle 影片、GPU 回可釋放。
- ⚠️ **idle ↔ 說話 接縫**是成敗點：idle 末帧頭部姿態要對上串流首帧 → codex 做「姿態錨定」（都用 source 中性姿態當錨）。兩引擎都要、生動更難（冷啟動慢）。

### A.2 audio→video 串流到 App
- **協定 WebRTC**（非 HLS/檔案）：即時雙向低延遲 + 抗抖 + 首幀 ~1.2s 是在此假設下。
- 管線：gpt-realtime-2 出音 chunk → AOL 餵 push_audio_chunk → 引擎吐 frames → WebRTC 推 iOS。
- 音畫同步：同 PTS 時間戳；生動若 frame 不穩、AOL 加 ~200ms jitter buffer。
- iOS：Capacitor + 原生 WebRTC plugin（新增工，引擎現無 iOS 串流接收端）。

### A.3 兩引擎熱插拔
- AOL session 狀態機，`current_engine` 可切。切換＝關 A stream → 開 B stream → WebRTC track replace（不重建連線）。
- **長輩端不重連、對話 context 不丟**（三顆腦在 AOL 上游、跟引擎解耦）。

### A.4 部署形態
- AOL：Modal FastAPI（scale-to-zero、**沐寧自己一套部署、不共用 Edward 個人引擎**）。
- Ditto worker：Modal serverless GPU、按秒、自動 scale-to-zero。
- LiveAvatar worker：⚠️ Modal 不一定餵得動 5×H800 多卡綁定 → 退路＝RunPod/雲廠自開 5×H800 + 輕量 controller 管租借/歸還/scale-to-zero。**標紅（風險 e）。**

## B · 切換流程 標準↔生動（含冷啟動容錯）

### B.1 切換時機 verdict：**「進聊聊那刻選」為主、「聊到一半切」為輔且異步預熱**
- 主路徑：尊榮進聊聊前選/記住偏好 → 選生動 → **idle loop 播放期間背景就 warmup 生動引擎** → 用 idle 把冷啟動藏起來（容錯 keystone c）。
- 不預設「隨時切」當主：冷啟動秒數未知，聊到一半才喚會乾等 = 災難。聊到一半切 = 異步預熱版（標準引擎先頂、好了無縫換、期間不中斷）。

### B.2 冷啟動容錯（keystone c 未知 → 三層防線、由實測秒數決定）
| 冷啟動實測 | 策略 |
|---|---|
| ≤5 秒 | idle loop 蓋過、長輩無感、直接進生動 |
| 5-20 秒 | idle loop + 過場語音、標準引擎先頂第一輪、好了無縫換 |
| >20 秒/失敗 | **降級**：本次用標準、背景續預熱、下輪再切 + 報警 |
- codex 必做：warmup 要 timeout + fallback。**永不准長輩盯轉圈等 GPU。「對話不斷」優先於「一定生動」。**

### B.3 切換視覺無縫
- 標準/生動是同角色同臉（鎖死）→ 切換是「同臉變生動」非換人。
- ⚠️ **未知風險(d)**：Ditto 和 LiveAvatar 兩套模型、同張 source 輸出的神態/色調/比例可能有細微差（兩畫家畫同人）→ 切換瞬間若跳變長輩會察覺 → Gate 2 女巫上線前比對一致性。**標必驗。**

## C · 計量強制（兩油箱 · 伺服器記帳 · 不可被前端騙 · 沙利曼 Gate 5 必審）

### C.1 鐵律：計量在 AOL 伺服器端、前端只是顯示器
- 前端永不持有真實餘額、只顯示 AOL 推下來的數字；扣點/降級判斷全在 AOL；前端傳的餘額一律不信。

### C.2 兩油箱帳本（elder_id 主鍵）
```
elder_id:
  monthly_grant_seconds   # 每月送（尊榮≈120分=7200秒）、月初重置、當月歸零
  monthly_grant_period    # 計費週期（判斷該否歸零）
  purchased_points        # 購買點數（永不過期）、1點≈15秒生動
  ledger[]                # 每筆扣帳流水（audit trail）
```
- 月送 vs 購買兩欄位物理分開、不可合成一數字。對應 RevenueCat：RC App User ID ↔ elder_id；購買走 RC Virtual Currency + consumable；月送由 AOL 依訂閱狀態發（RC Server Notification V2 通知續訂 → 發當月額度）。

### C.3 燒法順序（鎖死）
1. 先燒 monthly_grant_seconds（月送、當月歸零的先用） → 2. 燒完自動接 purchased_points → 3. 兩個都空 → 觸發自動降回標準寧寧。

### C.4 即時扣 + 軟硬雙閘
- 即時扣：生動串流期間每 ~5 秒從帳本扣、邊扣邊推餘額給前端。
- 軟閘：餘額 <60 秒 → 友善提示 + 通知子女可加購（不打斷）。
- 硬閘（歸零）：close_stream(生動)→open_stream(標準)→track 熱插拔 → **對話一秒不斷**、表情從生動變基礎、推播子女「已切回標準、可加購」。
- ⚠️ 扣帳粒度：**寧可少扣不可多扣**（多扣=客訴+消保）；歸零判斷用「下一扣帳點會變負 → 提前在這點降級」。

### C.5 防騙（沙利曼會問）
- 扣帳憑據 = GPU 實際消耗秒數（cost_meter 從引擎拿、非前端計時）；所有扣帳寫 ledger 流水；購買走 StoreKit2 收據伺服器驗證；月送綁 RC Server Notification。

## D · 多租戶分艙（單租戶引擎 → 多人 fork 最重的一塊）

### D.1 分艙鍵 = `elder_id` 貫穿所有層
記憶分艙 / 感知分艙（morning_brief、perception）/ 帳本分艙 / 串流分艙（WebRTC 隔離）/ 語音分艙（gpt-realtime session context 不串味）/ 守護分艙（危機偵測+通知對象）。

### D.2 改造重點（codex 工項）
- 記憶層：引擎 `memory/shared` 現為全域共用 → 改成每個 key 自動注入 elder_id scope、物理上互讀不到。隱私硬底線。
- 訪談/喜好/關係記憶同樣全 scope 到 elder_id。
- **隔離強度**：DAO 層**強制帶 elder_id、缺了就拋錯**，不靠每支 API 自己記得加。**任一支漏 elder_id 過濾 = 資料串戶 = 災難級事故。**
- 家庭帳號（標準1位/家庭2位）：一付費帳號掛 N 個 elder_id；**帳本（點數）在付費帳號層共用、記憶/感知/對話在 elder_id 層各自獨立**——這二層結構要設計清楚否則家庭版記憶串味。

### D.3 隔離 vs 成本
- 記憶/感知/對話 = 邏輯分艙（同套服務靠 elder_id 隔離）即可、不必每人一套基建。
- 唯一物理獨佔 = 生動引擎那組 5×H800（per-session、是運算獨佔非資料分艙）。

## E · 三顆腦 + gpt-realtime-2 接點
- **反射腦**（gpt-realtime-2 S2S 台灣中文、已 live）出音 chunk → AOL 同時 ① 餵 avatar 引擎生影格 ② WebRTC 推音給 App。音訊是源、影格跟著音訊生 → 拿到音才喚臉、音停臉回 idle。
- **管家腦**（感知/記憶）背景跑、產出已備 context 存 elder_id 分艙記憶；對話當下注入 gpt-realtime-2 session（深而不慢）。AOL 只負責 session 啟動時把該 elder_id 記憶餵進去。
- **守護腦**監聽對話流（危機詞/情緒/作息）→ 觸發走推播通知子女/1925/119（不經 avatar）。
- 接點工程注意：引擎腦 API 現成、但**都要加 elder_id 參數**（fork 主要改動面）；gpt-realtime-2 走 OpenAI 用量計費 → 上線前過 license/成本/zero-retention 帳。

## 🔴 上線前必驗閘門（3 未證 keystone + 2 衍生風險）
| # | 閘門 | 現況 | 不過退路 | 風險 |
|---|---|---|---|---|
| (a) | Ditto TRT 25fps + 首幀≤1.5s | PoC 只跑 12fps、TRT 未證 | 退「純語音+靜態臉、看臉才生片段」混合 | 🔴 高（標準引擎命脈、影響全部人）|
| (b) | LiveAvatar 5×H800 真出 45fps | **最沒測、連 PoC 都沒做** | 退「離線預生特殊片段」、生動降級為奢侈片段非即時 | 🔴🔴 極高（生動引擎整個立論）|
| (c) | 生動引擎冷啟動秒數 | **完全未知** | B.2 三層容錯、>20s 只能背景預熱+本輪用標準 | 🔴 高（決定切換體驗）|
| (d) | 兩引擎同臉輸出一致性 | 未比對 | Gate 2 女巫比對、差異大則加過渡遮掩或限同 session 不混用 | 🟡 中 |
| (e) | Modal 能否綁 5×H800 給單 session | 未確認 | 退自管 RunPod/雲廠 instance + controller | 🟡 中 |

## 🎯 最高風險排序（誠實、不藏）
1. **🔴🔴🔴 生動引擎（LiveAvatar 即時）立論未證且最貴**：keystone (b) 45fps 連 PoC 都沒做、疊 (c) 冷啟動 + (e) Modal 綁定 + per-session 獨佔 5×H800 成本結構 = 三未知集中單一引擎。**CTO 強烈建議：投入完整生動基建前、先做「LiveAvatar 5×H800 45fps + 冷啟動」PoC（像 Ditto 那晚）。商業設計已把成本框住（服），但「跑不跑得出 45fps、冷啟動多久」是物理問題、只有 PoC 能解。**
2. **🔴🔴 兩引擎熱插拔視覺無縫 + 計量降級瞬間切換**：track replace 時序 + 姿態錨定 + 音畫同步、細節魔鬼多、第一版易「臉跳一下」要反覆調。
3. **🔴 多租戶記憶分艙**：elder_id 過濾漏一支 API = 串戶 = 上新聞等級。工程不難但容錯率為零、沙利曼必逐支審。
4. **🟡 最不確定：冷啟動秒數**——沒人測過 5×H800 睡死到出 45fps 幾秒（8 秒？90 秒？）、不編數字、必實測。

## CTO 收尾
- 外殼（A/C/D/E）有把握能蓋對；Edward 要的「天生吃兩引擎、一次蓋對」用 `AvatarEngine` 介面就是正解、此架構決策卡西法簽。
- 但生動引擎現在蓋在 3 個未測物理問題上。**強烈建議：第 0 階段 LiveAvatar PoC 先燒幾百塊釘死 (b)(c)、再讓 codex 大規模動工**（否則蓋一個月發現出不了 45fps 整塊重來 = 真燒錢、憲法 v5.4.23）。外殼可先蓋（介面解耦、LiveAvatar 用 stub 也能整合測試）；「投入完整生動基建」扳機按下前先看 PoC。
