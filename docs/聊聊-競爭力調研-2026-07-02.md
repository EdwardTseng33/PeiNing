## 一句話總評
**大方向前瞻、執行深度落後。** 架構骨架（native 即時語音＋三腦解耦＋記憶/感知分離＋臉部分層）站在 2026 主流甚至偏前瞻；但每層實作比 SOTA 落後一到兩個身位，**記憶層與感知層是最明顯短板**。
- 架構設計（骨架）：**8/10**
- 實作成熟度（落地）：**4.5/10**
- 綜合競爭力：**6/10**（贏得了慢速/純文字老陪伴 App；記憶與主動性上目前贏不了 Nomi 這類專精對手）

## 逐面向

### 1. 即時語音（S2S）— 持平偏前瞻 🟢
- SOTA：Gemini 3.1 Flash Live（2026-03-26 發布，native S2S、90+ 語言可對話中切換、30 HD 語音、約 $0.018/分、延遲 300–500ms）vs OpenAI GPT-Realtime。編排層業界預設 **Pipecat / LiveKit Agents**（打斷 86% precision/100% recall）；開源可自架 Moshi / Step-Audio / Ultravox。
- 我們：押 Gemini Live 對（多語言＋成本＋中文最強選項之一）。
- 缺口：① 供應商鎖定、Live 仍 preview（政策風險，無自架退路）② **打斷是框架能力不是模型能力**（長輩語速慢、停頓多會吃虧）③ 台語品質查無一手數據。
- 建議：語音層抽成「可替換介面」（保留切 OpenAI/LiveKit 退路＋拿框架級打斷）；台語實測、備 fallback。

### 2. 長期記憶 — 落後 🔴（最大短板之一）
- SOTA：三範式定型 **Mem0（向量抽取，消費級首選，LongMemEval 93+）／ Zep-Graphiti（時序知識圖譜、事實有效期）／ Letta（分層上下文）**。陪伴標竿 **Nomi AI 四層記憶＋Identity Core（讀弦外之音、主動追問）**；Replika 2.0 有「記憶儀表板」給用戶查改。共識：**寫入不覆蓋、留多版本、檢索時定當下相關**。
- 我們：`memory_items`（有結構化欄位＋計分）方向對，但**缺向量召回、缺時序/衝突、缺遺忘衰減、缺主動追問回路**。
- 建議：**別自建記憶引擎——接 Mem0（或 Zep 補時序）**，保留我們的 schema 當上層業務語義；補向量→有效期→衰減；抄 Nomi 分層＋Identity Core、抄 Replika 記憶儀表板（正好對「子女買」的信任訴求）。

### 3. 情境感知 grounding — 落後（起點對）🔴
- SOTA：**tool-calling / MCP + grounding**（Google Maps Grounding Lite via MCP 錨定 3 億地點/天氣/路程；Search Grounding）。原則：精準、有範圍地投餵 context，不爆 token。
- 我們：「主題→所需感知源」的 plan 抽象對，但**多為 stub 沒落地**；Gemini Live 本身支援 function calling 卻沒接。
- 建議：把 perception plan **落成真的 tool-calling/MCP**（天氣＝中央氣象署/Maps、在地就醫交通＝Maps、時間節慶回診＝行事曆＋記憶），只在需要時最小注入；**守護腦也吃感知**（寒流/高溫→主動關懷）。

### 4. 會動的臉 — 持平偏落後 🟡
- SOTA：商業 HeyGen LiveAvatar / Tavus（<500ms）/ Anam（180ms）；開源 Ditto（2025，A100 首幀 385ms、RTF~0.9＝單卡幾乎只夠 1 路即時）、NVIDIA Audio2Face-3D（MIT、超省顯存多路併發，但是 3D 臉非寫真）；2026 新串流模型 SoulX-FlashHead 等已超越 Ditto。
- 我們：「2D 保底→Ditto 自架」分層降級思路務實正確，但 Ditto 已非最前沿，**自架 GPU 併發成本是殺手、商用授權要逐條核**。
- 建議：**臉部非核心競爭力、別過早燒 GPU**；先把 2D 表情做好；要寫真臉先用 HeyGen/Tavus API **驗證留存與付費意願**再自架；自架可評估 Audio2Face-3D（省顯存）。授權務必核。

### 5. 多腦協同（orchestration）— 持平偏前瞻 🟢
- SOTA：解耦子系統（perception/brain/planning/action/memory）＋ **orchestrator/router 明確分派優先權**；框架 OpenAI Agents SDK / Google ADK / Anthropic SDK / LangGraph。核心經驗：**精準 scoped context handoff**。
- 我們：三腦切法完全踩主流且聰明——即時（反射）/背景（管家）/安全（守護·決定論）解耦正確。
- 缺口：① 無顯式 router／優先權仲裁（守護觸發時誰先、管家注入會否拖慢反射）② context handoff 未必最小化（爆 token 風險）③ **守護腦純關鍵字太單薄**（隱晦憂鬱/失智混亂/跌倒求助召回堪憂）。
- 建議：加輕量顯式 router（守護＞反射即時＞管家背景）；scoped 最小注入；守護腦升級「**確定性關鍵字＋語義分類器雙層**」；不必引入重框架（即時對延遲敏感，手刻協同可，但規則要顯式、可測）。

## 三個最高優先（架構級）
1. **記憶接 Mem0（別自建）** — 投報最高、與 Nomi 差距最大；決定產品「像不像真的記得長輩」。
2. **感知從 stub 落成真的 tool-calling/MCP grounding**（天氣/在地/時間）並接進管家腦＋守護腦 — 「主動關心」與「安全預警」的事實底座，用 Live 現成 function calling、成本低體感大。
3. **降單點風險** — 語音抽成可替換介面（留 OpenAI/LiveKit 退路＋框架級打斷）＋三腦加顯式 router＋守護腦雙層；臉部先驗證留存再投 GPU。
