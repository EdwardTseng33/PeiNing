## 1. 整體架構 ＋ 功能（卡西法 CTO ＋ 霍爾 CPO）

### 🔴 最大死穴：引擎是單租戶（Edward 個人）架構
- 引擎記憶/感知是**單一全域檔**（`shared_{key}.json`、`shared_perception.json`），**無 user_id 維度**；auth＝Google OAuth＋Edward allow list；聲紋只認 Edward 一人。
- 直接共用 → **長輩記憶互串＝隱私事故**，不是 bug。
- 解：**fork 一套**、記憶/感知/聲紋/auth 四條全加 `elder_id` 維度＋換儲存後端（Firestore/Postgres，主鍵第一欄 `elder_id`）；id 由 server 從 token 解出、不信 client；gpt-realtime session instructions 按 elder 灌。**[codex]（卡西法先給 schema）**

### 🔴 三個 P0 回路全缺
1. **危機安全網（守護腦）**：規格 P0，現只有一句提示。要：長輩 SOS 入口＋被動偵測（硬訊號 HealthKit 跌倒/SOS 最可靠、軟訊號對話危機詞當輔助）＋三級升級（關心→通知家人→1925/119）＋子女端警報；走獨立通道、推播能破靜音（critical alert 需 Apple entitlement）。**[codex]**
2. **付費系統**：見 §2。
3. **真 iOS 推播（命脈「主動找長輩」）**：現在通知只是畫面卡片。Capacitor APNs＋後端排程（清晨問候/用藥/關懷/危機）。**[codex]**

### ⭐ 狀態頁裝置串接引導（Edward 點名）
- 現況：血壓/睡眠/步數全寫死假資料、標「來自 Apple 健康」但 0 串接；第一次打開會全空。
- 資料流：Apple Watch→iPhone HealthKit（裝置本地）→App 被動讀→加密上傳「子女授權欄位」（按 elder_id 分艙）→子女端讀。
- **授權嵌 onboarding 第 2.5 步**（裝機時子女幫長輩授權；最小授權：步數/心率/睡眠/跌倒 4 類）。
- **沒裝置分層降級**（多數 60+ 無 Watch）：L1 有 Watch 自動帶入／L2 只有 iPhone 讀內建計步／L3 全沒有→不顯示空數據、改「寧寧的觀察」（從對話推）＋引導子女配錶。**鐵律：健康數據是加分層、對話陪伴才是命脈，別讓沒數據開天窗。**
- 隱私：健康原始數據盡量留裝置、雲端只存授權彙總欄位、絕不進 LLM prompt、跌倒走事件通知不走持續上傳。**[codex] UI 由女巫/蘇菲定、HealthKit/藍牙交 codex**

### 用戶旅程 4 斷點（霍爾）
①無登入/帳號/家庭群組綁定 ②通知非真推播 ③**完全無付費畫面（最痛）** ④長輩無求救入口。

### 「先 PoC 才知道」死穴
gpt-realtime 台灣中文長輩語音真實延遲（無長輩 log）｜Modal cold start 9.6s 對長輩致命（保溫燒錢 vs 遮蔽動畫）｜HealthKit 後台同步可靠度。

## 2. 付費流程 ＋ 金流架構（蘇菲 sub CFO ＋ 沙利曼 Trust）

### 🔴 付費流程 0/10：原型 0 個付費畫面
完整旅程該有：子女付→14 天試用→選方案→扣款→續訂→訂閱管理（升降級/取消，Apple 必備）。全缺。**[codex]**
- 長輩產品轉換特性：鉤子打**子女端「媽媽真的有在用」**（試用價值總結）；試用 14 天**從長輩首次對話起算**（非裝機日）；流失點在「長輩沒開口」需救援流程。

### 💰 Apple 抽成重算（CFO ＋ Trust 調和後）
- iOS 訂閱**強制走 Apple IAP**（不能 Stripe 繞、繞＝下架；不需金流牌照——乾淨）。
- **抽成：申請「小型開發者方案」（年營收<US$1M 即 15%、我們符合、可立刻申請）→ 15%**；沒申請＝30%。

| 方案 | 標價 | 實收(15%) | 實收(30%) | 月成本 | 毛利率(15%) | 毛利率(30%) |
|---|---|---|---|---|---|---|
| 基本 | 299 | 254 | 209 | 150 | 41% | 🔴 28% |
| 完整⭐ | 499 | 424 | 349 | 150 | 65% | 🟡 57% |
| 深度 | 899 | 764 | 629 | 150 | 80% | 76% |

- **NT$299 是隱憂**（30% 下會虧、15% 下也偏薄、易被 API 成本吃掉）。
- 「65% 毛利」對投資人**首年要誠實修一檔**（取決於是否申請 SBP + 用量控管）。
- 財務模型要修 2 處：①抽成口徑（標 SBP 要申請、否則 30%）②試用流程（見下）。

### 金流架構（沙利曼）
- **B2C＝Apple IAP**：StoreKit2＋**伺服器端憑證驗證**（不只信 App 端）＋**Server Notifications V2 webhook**（接退款/取消/帳單失敗）＋entitlement 服務＋沙盒測試。**[codex]**
- **IAP plugin 選型**：建議 **RevenueCat 起步**（Capacitor 官方 plugin、收據驗證/webhook/對帳全包、免費額度夠初期、不碰卡號）。**釐清：RevenueCat 是「訂閱狀態中台」非金流通道，不違背「全自架」初衷（初衷指不付每分鐘語音費）。** 規模化再轉自建。
- **跨裝置授權**：訂閱**綁「沐寧帳號」不綁 Apple ID**——子女 Apple ID 付款→entitlement 記在長輩沐寧帳號→長輩登入解鎖（onboarding「給誰用/邀家人」正好接）。**[codex 帳號模型先設計 付費者↔受益長輩 綁定]**
- **試用流程要改**：財務模型「前 3 天不用卡、第 4 天才要卡」**IAP 做不到**；改 **14 天 IAP Introductory Offer（啟動即綁卡、到期自動轉）**，轉換率影響極小、一次過審。
- **B2B2C 第二條金流**（長照/保險、Year 2）：合約/請款/Stripe/台灣電子發票，與 B2C 分流；**entitlement 服務一開始就設計成「授權來源可多種」**（`source: apple_iap|b2b_contract|promo`）免日後大改；B2B 授權走「企業帳號登入即享權益」非「App 內輸序號解鎖」（避踩 Apple 紅線）。

### 🔴 Apple 審核金流紅線（會卡審、上線前必補）
①訂閱揭露三件套（價格/週期/取消＋隱私政策＋EULA 連結，缺＝退件第一名）②醫療宣稱掃描（零「治療/診斷/醫療級」、用藥呈現為鬧鐘非指導）③HealthKit 用途揭露＋健康資料同意（含失智法定代理同意）④試用轉收費揭露。

### 合規（沙利曼可立刻動手起草）
金流架構文件｜上架前 checklist｜隱私政策+服務條款底稿（健康特種個資同意、語音出境 OpenAI 告知、訂閱條款）｜B2B2C「Trust 一頁說明」（資料流+子處理者清單+加密+出境）｜DPA 模板｜OpenAI zero-retention 條款確認。

## 3. voice / avatar 完善度 ＋ 接續（卡西法 CTO）

### 語音（gpt-realtime-2 台灣中文）🟡
- 已有：引擎 `/sdp` WebRTC proven、server VAD（自然打斷）、persona 注入。
- 缺口：**Capacitor WebView 內 getUserMedia 麥克風橋接能不能穩＝頭號未知**（iOS WKWebView WebRTC 有坑）；多租戶 session 按 elder 注入；原型是假聲。
- 接續：①**[codex] Capacitor WebRTC 麥克風橋接 PoC（最高風險先驗）** ②[codex] 沐寧自己一套 Modal 部署（剝 Edward allow list、改多租戶）③[codex] session 按 elder 注入記憶 ④真機台灣中文長輩延遲實測 ⑤cold start 對策。

### 擬真 avatar（Ditto）+ 2D 🔴
- 已有：Ditto 單卡 PoC 跑得動（24GB 夠、Apache-2.0 可商用）。
- 從「跑得動」到「即時服務」缺：①TRT 真即時**還沒驗**（12→25fps 是預期非實證）②**雲端常駐 GPU 服務 vs Modal scale-to-zero 成本矛盾**（即時要 GPU 一直開、燒錢）③嘴型串 gpt-realtime 即時音流對齊 ④App 端影像接收架構 ⑤多路並發成本（毛利假設押這條、現在是空的）。
- 接續：①**[Edward 花 GPU 錢] 先驗 TRT 真即時＝go/no-go 閘門**（過不了 v1 直接砍擬真、只上 2D）②[codex 卡西法先設計] 即時 GPU 服務 PoC ③[codex] 嘴型串流對齊 ④[codex] App 端接收 ⑤**2D 軌保底（Live2D＋Audio2Face）＝v1 該先穩的**。
