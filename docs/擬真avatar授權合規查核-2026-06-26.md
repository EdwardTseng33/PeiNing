## 1. Ditto —— 🟢 能商用（PoC 首選）

**Repo（釐清）：** 霍爾盤點寫「ant-research」，實際 repo 是 **`github.com/antgroup/ditto-talkinghead`**（ACM MM 2025）。權重在 HuggingFace `digital-avatar/ditto-talkinghead`。同一專案、不影響判斷，但卡西法 clone 時認這個 URL。

| 查核項 | 結果 | 證據 |
|---|---|---|
| **授權類型** | **Apache 2.0** | README 原文："This repository is released under the Apache-2.0 license as found in the LICENSE file." |
| **能否商用收費** | ✅ **可以** | Apache 2.0 明確允許商用、修改、再散布 |
| **附加條件** | 僅 Apache 2.0 標準義務（保留 LICENSE + NOTICE + 標註修改處）| 無署名強制、無開源回饋、無收入分成 |
| **用量天花板** | ✅ **無** | README / LICENSE / HF model card 三處皆無 MAU / 用戶數 / 營收門檻 |
| **碼 vs 權重授權是否分開** | ✅ **同一個 Apache 2.0、未分離** | HF model card 明寫權重同 repo Apache 2.0；提供 PyTorch(.pth)/ONNX/TensorRT 三種格式、皆同授權。**沒踩「碼開源、權重限非商用」的常見地雷。** |
| **資料是否回傳** | ✅ **不回傳** | HF README 實查：純本機安裝（conda/pip + 本地 checkpoint 目錄），無遠端 API 呼叫、無 telemetry、無線上啟用 |
| **acceptable use 條款** | ✅ **無額外限制** | 不像 Wan，Ditto 文件未附倫理使用條款 |

**沙利曼判斷：** 授權面**三項全綠、最乾淨**。這是你要我分辨的「碼/權重是否分開授權」測試裡唯一完全沒踩雷的——碼和權重綁同一個 Apache 2.0。對沐寧（商業收費 App）沒有任何授權阻礙。**唯一剩下的不確定是技術面（RTX 4090 真實 fps），那是卡西法 PoC 的事、不是授權問題。**

> ⚠️ **PoC 第 0 件事（紀律）：** clone 下來後，請工程的人**親眼確認 repo 根目錄 `LICENSE` 檔確實是 Apache 2.0 全文**、且 NOTICE 檔（若有）無附加限制。我讀的是 README 轉述 + HF model card，已高度一致，但「商用上線」這種事，原檔要有人用眼睛掃過一次——這是死路備忘錄「接任何模型先讀 README + LICENSE」的紀律。

## 2. LiveAvatar 14B —— 🟡 有條件能商用（備選、非授權問題卡關）

**Repo：** `github.com/Alibaba-Quark/LiveAvatar`（阿里 Quark · ECCV 2026）。base 模型 = `Wan-AI/Wan2.2-S2V-14B`。

> ⚠️ **務必別跟 HeyGen 的 liveavatar.com 搞混**（那是付費雲端、零開源）。這裡查的是阿里開源同名專案。

| 查核項 | 結果 | 證據 |
|---|---|---|
| **授權類型（碼）** | **Apache 2.0（標準全文、無附加）** | LICENSE 原檔實讀：standard Apache 2.0、Sections 1-9 + 官方 appendix、**無任何 modified clause**。README："The majority of this project is released under the Apache 2.0 license." |
| **授權類型（base 權重）** | **Apache 2.0** | README："The Wan model (Our base model) is also released under the Apache 2.0 license." HF `Wan2.2-S2V-14B` License 欄位確認 Apache 2.0。 |
| **能否商用收費** | ✅ **可以** | 碼＋權重皆 Apache 2.0 |
| **「majority」的 minority 是什麼？** | ⚠️ **README 措辭含糊、但 LICENSE 原檔是乾淨 Apache** | README 用「majority」這個字（通常是地雷信號），但我去讀 LICENSE 原檔＝標準 Apache 無附加；推測「minority」指的是 base Wan 模型被當獨立元件列、而它本身也是 Apache 2.0。**目前查到的證據裡，沒有任何一塊落在 non-commercial 或 custom license。** |
| **碼 vs 權重授權是否分開** | ✅ **分開列、但兩邊都 Apache 2.0**（碼一個 LICENSE、Wan base 一個 LICENSE，皆 Apache）| 這是「分開但都乾淨」、不是「碼開權重鎖」的地雷 |
| **用量天花板** | ✅ **無** | 無 MAU / 用戶數 / 營收門檻 |
| **acceptable use 條款** | ⚠️ **有（base Wan 附倫理使用條款）** | Wan2.2 model card 原文："You are fully accountable for your use of the models, which must not involve sharing any content that violates applicable laws, causes harm to individuals or groups, disseminates personal information intended for harm, spreads misinformation, **or targets vulnerable populations**." |
| **「research preview」字眼** | ⚠️ **是研究預覽** | README 自稱 "a research preview"、要求發現違規請聯絡作者（jmliu1217@gmail.com）。商用前心裡有數：這是研究釋出、非商業產品保證。 |

**沙利曼判斷：** 授權面**可商用**，碼＋權重都是真 Apache 2.0、無天花板。需要在意的兩點：

1. **acceptable use 的「targets vulnerable populations」（針對脆弱族群）** —— 沐寧服務長輩＝脆弱族群。**這條不是禁止你服務長輩，是禁止你拿模型生成「傷害/詐騙/針對脆弱族群的惡意內容」。** 沐寧是正當陪伴用途、不違反；但這條提醒一件事：**沐寧若用擬真人臉，務必避免「冒充真實親人詐騙長輩」的風險場景**（這也是我下面合規段會強調的恐怖谷/冒名風險）。倫理條款本身對正當用途無實質阻礙。
2. **真正卡關的不是授權、是硬體** —— 霍爾盤點 + 卡西法成本帳已釘死：即時 45fps 要 5×H800、單卡要 48-80GB，RTX 4090 跑不動即時。**LiveAvatar 的問題是「貴」不是「不能用」。** 授權我放行，但這條只在「Edward 願吞 GPU 成本」時才值得碰。

> ⚠️ **「research preview」的 caveat：** 研究預覽代表作者不保證 API 穩定、不保證長期維護。商用押注在它身上前，卡西法要評估「若上游不再維護、我們自己能不能 fork 養」——Apache 2.0 給你 fork 的權利，這點無虞。

## 3. Duix-Mobile —— 🔴 不建議當商業主力（1000 MAU 硬天花板）

**Repo：** `github.com/duixcom/Duix-Mobile`（硅基智能 · 端側即時 SDK）。

霍爾盤點懷疑「滿 1000 MAU 觸天花板」——**我讀了 LICENSE 原檔，確認屬實，而且比想像中更綁。**

| 查核項 | 結果 | 證據（LICENSE 原文） |
|---|---|---|
| **授權類型** | 🔴 **不是 Apache/MIT，是 "duix.com Community License"** | 這是類 Llama Community License 模式的自訂授權、不是標準開源授權 |
| **能否商用收費** | ⚠️ **1000 MAU 以下可以、以上要回求授權** | "If...the monthly active users of the products or services made available by or for Licensee...is greater than **1 thousand monthly active users** in the preceding calendar month...you must request a license from duix.com" |
| **天花板數字** | 🔴 **1,000 MAU（純用戶數、無關營收）** | 門檻是「上個自然月 > 1000 月活躍用戶」即觸發 |
| **超過天花板後** | 🔴 **須對方「sole discretion」才給授權** | 超過 1000 MAU 後，未經 duix.com 明確另行授權不得行使權利，而對方「in its sole discretion」決定給不給——**等於把你的商業命脈交在對方手上** |
| **強制署名** | 🔴 **須顯著標「Powered by Duix.com」** | "prominently display 'Powered by Duix.com' on a related website, user interface, blog post, about page, or product documentation" |
| **模型命名強制** | 🔴 **衍生模型名須冠 "duix.com"** | "include 'duix.com' at the beginning of any such AI model name" |
| **強制保留 notice** | 🔴 須保留 | "duix.com is licensed under the duix.com Community License, Copyright © duix.com Platforms, Inc. All Rights Reserved." |
| **碼 vs 權重授權是否分開** | ⚠️ Community License 涵蓋 materials（碼＋模型一起綁同一份限制）| 不是「分開」、是「一起被綁」 |

**沙利曼判斷：** **這是三個裡唯一有商業地雷的。** 對沐寧（目標就是要養出一群長輩付費用戶）來說：

- **1000 MAU 是「成功就觸雷」的天花板** —— 沐寧只要做起來、過 1000 個長輩，就必須回求 duix.com 授權，而對方有完全裁量權（可以不給、可以開高價、可以附條件）。**等於把產品的天花板和議價權拱手讓給上游一家中國公司。** 這是商業模式層的結構性風險，不是技術問題。
- **強制掛「Powered by Duix.com」** —— 一個對長輩主打「信任、貼心管家」的品牌，App 裡被迫掛中國廠商 logo，品牌調性 + 地緣觀感雙重不利。

**結論：可以拿來當「端側即時」的技術參考 / 學架構，但不要當商業主力引擎。** 這跟霍爾、卡西法的判斷一致，我從授權原文再確認一次：**不是「有條件能」，是「成功就會被掐脖子」**——對一個要長期經營付費用戶的產品，這條紅線我畫實。

## 4. 中國來源風險（三者共通 · 對台灣商業產品的合規視角）

三個方案全是中國國籍（antgroup / 阿里 / 硅基智能）。我分四個維度看，對沐寧（台灣商業 iOS App、面向長輩）：

### 4.1 資料是否回傳 — 🟢 低風險（皆本機 inference）

- 三者都是**下載權重後本機跑的 talking-head 模型、不是雲端 API**。Ditto 我實查確認無遠端呼叫、無 telemetry、無線上啟用；LiveAvatar/Wan 同屬本機 inference 路線。
- **關鍵守則：** 模型權重下載完、跑在你自己的機器/你租的 GPU 上 = 長輩的語音/影像資料**不出你的伺服器、不回中國**。這是「自架」相對「串中國雲端 API」最大的合規優勢。
- ⚠️ **唯一要盯的：** 部分中國 repo 會在安裝腳本/推理鏈裡從中國 CDN（如 ModelScope）抓權重或依賴。**這是「下載階段」的連線、不是「服務運行階段」的資料回傳**——但卡西法 PoC 時請留意：① 權重來源盡量走 HuggingFace（非中國 CDN）；② 確認執行期沒有對外連線（可用防火牆/抓包驗證一次）。確認後，運行期資料落地台灣/你的機器，無回傳疑慮。

### 4.2 地緣 / 合規對台灣產品 — 🟡 中性偏低（自架已大幅降風險）

- **自架 = 把地緣風險從「服務依賴」降到「碼/權重依賴」。** 你不依賴中國公司的線上服務（不會被斷線/被審查/被要求交資料），只是用了它開源的權重檔。Apache 2.0 模型（Ditto/LiveAvatar）一旦下載，對方無法事後撤回你的使用權。
- **Duix-Mobile 例外**：它的 Community License 讓你「成功後須回求授權」＝仍對中國公司有持續依賴關係＝地緣風險未解除。這是它比另兩個差的又一理由。
- **對長輩個資的合規**（台灣個資法 / 未來若上架要符合 Apple 隱私規範）：自架本機 inference 對「資料不離境」有利；真正要做足的是**沐寧 App 本身的個資告知同意 + 資料最小化 + 加密儲存**，那是產品層的事、與選哪個模型無關（我未來做產品 Gate 0 時會一起看）。

### 4.3 專利地雷 — 🟡 無法在初篩排除（但 Apache 2.0 已給專利授權）

- **好消息：** Apache 2.0 **內含明示專利授權條款**（貢獻者授予你使用其專利的權利）——這是 Apache 比 MIT 對商用更友善的關鍵。用 Ditto/LiveAvatar，貢獻者自己的相關專利你有授權。
- **無法排除的：** 第三方（非貢獻者）的 talking-head / lipsync 相關專利侵權風險，**任何開源模型都無法在初篩階段排除**，這是整個生成式 avatar 領域的共通不確定性、不是這三個方案獨有。真正要在意這個，是在「產品規模大、有被告價值」時做 FTO（freedom-to-operate）專利檢索——那是法務 + 專利律師的工作、不是現在 PoC 階段該擋的事。
- Duix-Mobile 的 Community License **不像 Apache 那樣明確給專利授權**（自訂授權的專利條款通常更模糊）——又一個它較不利的點。

### 4.4 對「擬真人臉 + 長輩」的特有風險 — ⚠️ 產品設計層要防（非授權問題，但我必須提）

這條超出純授權、但屬我 Head of Trust 職責、且 Wan 的 acceptable use 也點到「脆弱族群」，必須講：

- **冒名詐騙風險：** 擬真人臉若做得像真實親人，存在「被濫用冒充家人詐騙長輩」的場景。**這不是模型授權禁止你做沐寧，是提醒沐寧的產品設計要內建防線**——例如：擬真形象用「明確非特定真人」的虛擬臉、不提供「上傳家人照片生成擬真臉」功能（這功能本身就是詐騙溫床）、App 內明示「我是 AI 管家、不是您的家人」。
- **恐怖谷 / 長輩接受度：** 霍爾已分析（2D 可愛動物可能比擬真人臉更適合長輩、PARO 邏輯）。從信任角度我同意——**擬真做不好反而傷信任**。
- → 這些是**未來沐寧產品 Gate 0（design 階段資安/合規架構 review）**我會正式跑的項目。**現在 PoC 階段不阻擋**，但先記在這裡，等真要把擬真臉做進產品時拿出來。
