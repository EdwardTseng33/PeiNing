## 一、記憶層評估

### 1.1 對標 Mem0（真讀懂它的技術框架）
Mem0 的核心不是「一個向量庫」，是 **兩階段 LLM 流程：Extract（萃取）→ Update（對帳）**。Update 階段對每條新事實撈語意最近的既有記憶，用 tool-call 讓 LLM 選 **ADD / UPDATE / DELETE / NOOP**——**寫入的當下就把記憶對帳乾淨**，不是無腦堆、也不是事後補救。它的準確度紅利主要來自「寫入乾淨」，不是花俏檢索（LOCoMo J 66.9%，token 只用塞爆 context 的 ~1/15）。圖變體 Mem0g 加 entity/relationship 圖，temporal 題最強，但 latency/token ×2。

### 1.2 我們 vs Mem0（誠實）
- **設計層：夠、且比 Mem0 更貼長輩陪伴。** 4 層壽命記憶（永久核心 vs 每日刷新）、活的側寫（Mem0 沒有）、健康永遠記＋記趨勢、中文台灣在地、敏感標記＋同意分層——都是 Mem0 給不了的產品貼合度。**Edward「不接 Mem0 依賴、只借做法」的判斷正確。**
- **實作層先前的最痛缺口：記憶只增不修。** `supersedes_memory_id`／`valid_until` 是死欄位，沒有任何流程會設值 → 搬家、改名、病況變化會「新舊兩版並存」→ 踩爛北極星「不抱過時的你 / 不自打嘴巴 / 不像失智」。
- **交叉驗證**：記憶驗收測試第 9 條（時序/衝突）獨立測出同一破口 → 黑箱測試與白箱 Mem0 分析指向同一件事。

## 二、感知層評估（記憶完善後盤）

### 2.1 一句話
感知層目前是 **「計畫書＋空抽屜」**：系統能用關鍵字判斷「這輪該查天氣/地圖/新聞」，但**沒有任何一個真實資料源被接上**——沒有真天氣、沒有真時間、沒有地圖、沒有每日排程、沒有即時語音查事實。設計的 7 感知源，落地為真實工具呼叫的是 **0 個**。**完成度約 12–15%。**

### 2.2 兩個致命斷點
1. **孤兒資料表**：`perception_snapshots` 存取 API 齊全，但 `build_reply_context` **從不讀回**它 → 就算存了真實事實也永遠進不了回話。
2. **假天氣寫死**：`chat_engine.py` 一句 hardcoded「今天轉涼有寒流」冒充感知。

### 2.3 缺口清單（可出貨長輩陪伴的門檻）
- **P0（不補第一天就穿幫）**：① 真時間/時區注入（幾乎免費、最該先做）② 真天氣接中央氣象署 CWA ③ 打通 snapshot→回話斷點 ④ 感知餵守護腦（寒流→保暖提醒）。
- **P1**：⑤ 每日感知排程（清晨簡報→主動開口引擎）⑥ 當前檢索真的去查（新聞/影劇/財經擇一）⑦ 即時語音掛 function-calling。
- **P2**：⑧ 地圖/行事曆/家人狀態 ⑨ 智慧鏡頭 ⑩ 興趣圖譜併入即時層＋快取自動過期。

### 2.4 感知層 VERDICT
設計完整前瞻，但**實作接近空白**。地基與接口都預留好了（endpoint/schema/plan/prompt 插槽），差的是「連到真實世界」——真實外部資料源連線數 = 0。**P0 四件是最小可信集合，其中真時間幾乎免費。**

## 三、給 Edward 的結論
1. **記憶大腦**：設計贏 Mem0 的貼合度、實作補完最痛的「寫入即對帳」後約 95%，再收一個 recency 衰減就能誠實喊 100%。
2. **記憶驗收**：11 條標準、會自動跑、目前全過——之後每次改記憶都能一鍵驗收不回退。
3. **感知層**：目前約 13%、是下一個主戰場；P0 從「真時間＋真天氣＋打通斷點＋餵守護腦」起手。
